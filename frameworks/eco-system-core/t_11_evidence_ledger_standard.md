# T11 — Evidence Ledger Standard

## Document Status
**System:** Market-Intelligence Operating System  
**Layer:** Companion Governance Layer  
**Designation:** T11  
**Document Type:** Standard  
**Status:** Canonical Draft  
**Purpose Class:** Traceability / Evidence Governance / Audit Control

---

## 1. Purpose

The **Evidence Ledger** is the authoritative evidence-governance artifact for the Market-Intelligence Operating System.

Its function is to create a single, structured record of the sources, claims, confidence boundaries, contradictions, and freshness conditions that support all downstream analytical work.

The ledger exists to ensure that the operating system does not merely produce compelling analysis, but produces analysis that can be:

- traced
- challenged
- refreshed
- defended
- revised
- compared over time

The Evidence Ledger is the system’s control layer for factual grounding.

---

## 2. Strategic Role in the Operating System

The Market-Intelligence Operating System already produces analytical outputs that depend on disciplined sourcing, confidence boundaries, and explicit assumptions. The Evidence Ledger operationalizes those requirements by turning evidence management into a governed system rather than an informal research habit.

It serves five strategic roles:

1. **Traceability** — every meaningful claim should be traceable to an identified evidence source or explicitly marked as inferential.
2. **Auditability** — reviewers must be able to verify what evidence was used, when it was gathered, and how strong it is.
3. **Freshness control** — the system must know which conclusions are vulnerable to staleness.
4. **Contradiction handling** — conflicting evidence must be surfaced and governed, not silently collapsed.
5. **Cross-document continuity** — evidence used in EA, T1–T10, and later companion documents must be linkable across the full lifecycle.

This document is therefore a **governance backbone** rather than a narrative analysis template.

---

## 3. What This Document Is

This document is:

- a structured ledger for research evidence
- a control mechanism for source quality and claim support
- a freshness and contradiction registry
- a cross-reference layer between sources and analytical outputs
- a reusable artifact that persists beyond any single report

---

## 4. What This Document Is Not

The Evidence Ledger is **not**:

- an ecosystem analysis
- a market summary
- a bibliography without structure
- a notes dump
- a quote repository without claim linkage
- a substitute for judgment
- a place to hide unsupported assertions

It does not perform the analysis itself. It governs the evidence used to produce analysis.

---

## 5. When to Commission or Use It

The Evidence Ledger should be created at the start of any formal ecosystem research effort and maintained continuously throughout the lifecycle of that effort.

It should be mandatory whenever any of the following are true:

- a formal Ecosystem Analysis is being produced
- one or more Intelligence or Strategy templates are being commissioned
- multiple analysts or agents are contributing to the same body of work
- the work will inform strategic, capital, platform, or product decisions
- conclusions may need to be refreshed later
- the analysis may be audited or red-teamed

It should not be deferred until the end.

The ledger must be built alongside the work, not reconstructed after the fact.

## 6. Mandatory vs Conditional Use

**Mandatory** for:

- Ecosystem Analysis Standard work

Even in conditional cases, the ledger is recommended where practical.

## 8. Outputs Produced

The Evidence Ledger produces the following outputs:

- a governed inventory of sources
- explicit claim-to-source linkage
- source quality classifications
- contradiction records
- freshness and staleness signals
- evidence coverage diagnostics
- unresolved evidence gaps
- reusable audit trail for later review or refresh

These outputs should support every later document in the system.

## 9. Position in Sequencing Logic

T11 should be generated **before** or **at the same time as** formal analysis begins.

It sits in the sequence as follows:

1. opportunity or scope defined
2. evidence collection begins
3. **T11 Evidence Ledger established**
4. EA and selective T1–T10 work proceeds
5. T12 assumptions and unknowns formalized
6. T13 decision compression created
7. post-T10 conversion and validation work proceeds
8. refresh and learning workflows update the ledger continuously

The ledger is therefore both an early-stage and persistent artifact.

## 11. Core Analytical Questions

The Evidence Ledger should be designed so that a reviewer can answer the following questions quickly:

1. What evidence supports this claim?
2. Is the evidence primary, secondary, observational, inferential, or speculative?
3. How current is the evidence?
4. How credible is the source in this context?
5. Are there contradictory sources?
6. Has the contradiction been resolved, deferred, or left open?
7. Which sections of the operating system depend on this evidence?
8. Which conclusions are highly evidence-backed, and which are more interpretive?
9. What information is missing?
10. What would need to change for this conclusion to be revisited?

## 12. Required Structure of the Ledger

The Evidence Ledger should contain the following core sections.

### 12.1 Header
- ecosystem name
- scope name or project name
- version number
- research window
- last updated date
- owner / maintainer
- status

### 12.2 Source Registry
A structured list of all sources used or considered.

### 12.3 Claim Linkage Table
A table that maps claims to supporting evidence.

### 12.4 Contradiction Register
A table for evidence conflicts, inconsistencies, or disputes.

### 12.5 Freshness Monitor
A section that flags time-sensitive evidence and expected review thresholds.

### 12.6 Coverage Diagnostics
A summary of what is well-supported, weakly supported, missing, or over-dependent on thin evidence.

### 12.7 Evidence Gap Log
A registry of unanswered questions or claims that remain under-supported.

### 12.8 Change Log
A record of ledger updates over time.

## 14. Source Classification Rules

All sources must be classified in a way that allows the system to distinguish strong evidence from weak evidence.

### 14.1 Primary Sources
Examples include:
- official platform documentation
- regulatory filings
- company earnings materials
- first-party policies
- technical specifications from the issuer
- official marketplace requirements

Primary sources should be preferred wherever available.

### 14.2 Secondary Sources
Examples include:
- reputable analyst commentary
- industry publications
- trusted market research summaries
- credible press reporting used to interpret a primary event

Secondary sources may support context, synthesis, and external interpretation, but should not quietly replace primary evidence when primary evidence exists.

### 14.3 Tertiary or Low-Control Sources
Examples include:
- uncontrolled commentary
- hearsay summaries
- reposted claims without traceable origin
- community statements without verification

These may occasionally surface leads, but they should not carry core analytical load unless explicitly marked and justified.

---

## 15. Authority Rating Framework

Each source must receive an authority rating appropriate to context.

Recommended scale:

- **A — Authoritative:** direct, official, highly relevant, and current
- **B — Strong:** credible and relevant, but not fully canonical
- **C — Useful but limited:** partially credible, contextual, or incomplete
- **D — Weak:** low-confidence, weakly attributable, or highly indirect
- **E — Unacceptable:** should not be used for support

Authority must be judged relative to the claim being supported.

A source can be A-level for one question and C-level for another.

---

## 16. Freshness Rating Framework

Because some evidence decays faster than others, each source should receive a freshness rating.

Recommended categories:

- **F1 — Highly time-sensitive**
- **F2 — Moderately time-sensitive**
- **F3 — Slowly changing**
- **F4 — Structurally stable**

Examples:
- policy updates, pricing, roadmap clues, and marketplace rules may be F1
- annual reports may be F2 or F3 depending on use
- foundational ecosystem structure may be F3 or F4

Freshness must be recorded independently from authority.

A source may be authoritative but stale.

---

## 17. Support Strength Rules

Claims should not all be treated equally. The ledger must distinguish support strength.

Recommended categories:

- **Directly supported** — clearly established by source evidence
- **Strongly supported** — not explicit in one source, but strongly substantiated by multiple sources
- **Conditionally supported** — plausible but dependent on assumptions or incomplete evidence
- **Weakly supported** — thin evidence; use caution
- **Unsupported** — no acceptable support found

Unsupported claims should not appear in strategic outputs unless explicitly framed as hypotheses.

---

## 18. Inference Level Rules

The ledger must mark whether a claim is evidence-derived or analyst-constructed.

Recommended categories:

- **I0 — Descriptive fact**
- **I1 — Low inference**
- **I2 — Moderate inference**
- **I3 — High inference / synthesis**
- **I4 — Strategic hypothesis**

This is necessary because many of the system’s most valuable outputs are not simple facts; they are judgments. The ledger should not suppress judgment. It should govern it.

---

## 19. Confidence Handling

Confidence should reflect the relationship between evidence strength, contradiction level, and inference load.

Recommended fields:
- high confidence
- moderate confidence
- provisional confidence
- low confidence

Confidence should not be assigned based on writing quality or analyst certainty. It must reflect evidentiary condition.

---

## 20. Contradiction Handling Standard

Contradictions must be recorded whenever:

- two sources disagree materially
- one source appears to supersede another
- evidence differs by geography, segment, or time frame
- the interpretation of a source is contested
- platform or policy language is ambiguous enough to support multiple readings

Contradictions should never be quietly ignored.

Each contradiction must be assigned one of the following statuses:

- resolved
- unresolved but low impact
- unresolved and decision-relevant
- pending additional evidence

Any unresolved contradiction that affects T10, T13, T14, T15, T17, or T18 should be escalated visibly.

---

## 21. Evidence Gap Standard

A formal gap should be logged whenever:

- an important question lacks acceptable evidence
- a key conclusion depends on a thin base of support
- evidence exists but is outdated relative to decision stakes
- access constraints prevented collection of needed evidence
- the ecosystem is shifting too quickly for older evidence to remain reliable

Gaps are not failures. Ungoverned gaps are failures.

---

## 22. Required Usage Rules

The following rules are mandatory.

### Rule 1 — No silent sourcing
No strategic claim should appear in formal outputs without either:
- linked evidence, or
- explicit labeling as inference, assumption, or hypothesis

### Rule 2 — No hidden contradictions
Contradictory evidence must be logged.

### Rule 3 — No stale evidence masquerading as current evidence
Time-sensitive evidence must be freshness-scored.

### Rule 4 — No single-source fragility for major conclusions
Major rankings, vectors, or platform judgments should not depend on one weak source where broader evidence is available.

### Rule 5 — No undifferentiated source lists
A simple bibliography is insufficient. Source function and support role must be explicit.

### Rule 6 — No evidence laundering through summary language
An unsupported inference cannot be made to look stronger by being stated elegantly.

---

## 23. Recommended Operating Procedure

### Step 1 — Register sources as they are encountered
Do not wait until the report is finished.

### Step 2 — Assign source classifications
Record authority, freshness, modality, and relevance.

### Step 3 — Link claims to sources continuously
As analytical writing develops, update the claim linkage table.

### Step 4 — Record contradictions and gaps immediately
Do not defer conflict logging.

### Step 5 — Review evidence coverage before major decisions
Prior to T10 or T13 release, examine weakly supported areas.

### Step 6 — Update ledger during refresh cycles
When the market changes, the ledger should be revised before or alongside narrative documents.

### Step 7 — Use the ledger in learning capture
When real-world outcomes diverge from prior conclusions, update the evidence trail.

---

## 24. Quality Standard / Acceptance Criteria

The Evidence Ledger should be considered acceptable only if all of the following are true:

1. All strategically meaningful claims in formal outputs can be traced to evidence or clearly marked inference.
2. Sources are classified by type, authority, and freshness.
3. Contradictions are visible and governed.
4. Evidence gaps are explicitly logged.
5. Major conclusions do not rely on unexamined source fragility.
6. The ledger is structured enough to support red-team review.
7. The ledger is reusable during refresh and post-launch learning.
8. A reviewer can determine which parts of the system are strongly grounded and which remain provisional.

---

## 25. Failure Modes and Misuse Warnings

Common failure modes include:

### 25.1 Bibliography masquerading as governance
A long source list without claim linkage is not a ledger.

### 25.2 Post hoc reconstruction
If the ledger is built after conclusions are finalized, evidence discipline will degrade.

### 25.3 Overweighting official sources without context
Official sources may be authoritative about policy, but not always about market experience, behavioral reality, or practical friction.

### 25.4 False precision
A detailed ledger does not guarantee correct conclusions.

### 25.5 Freshness blindness
Even excellent evidence decays.

### 25.6 Inference concealment
Analyst synthesis must be marked, not smuggled in as direct fact.

### 25.7 Non-maintenance
A ledger that is not updated becomes a stale compliance artifact rather than a live governance system.

---

## 26. Source and Evidence Expectations

The ledger should follow these evidence preferences:

1. use primary evidence first where available
2. use credible secondary evidence to contextualize, compare, or triangulate
3. mark tertiary or low-control evidence clearly
4. separate observation from interpretation
5. preserve retrieval dates for all volatile sources
6. record when a source has been superseded or invalidated

Where evidence is unavailable, this fact should be recorded explicitly rather than patched over with speculation.

## 28. Relationship to Agentic Research Systems

The Evidence Ledger should be designed for both human and agentic operation.

This means:

- fields should be discrete and machine-legible where possible
- claim IDs and source IDs should be stable
- contradiction and gap states should be enumerable
- retrieval dates should be standardized
- evidence linkage should be updateable incrementally

An agent should be able to contribute to the ledger without eroding evidentiary discipline.

---

## 29. Versioning Notes

The Evidence Ledger Standard should evolve when any of the following occur:

- repeated red-team reviews expose a structural weakness in evidence governance
- new source types become strategically important
- the operating system expands into new ecosystem categories
- contradiction handling proves insufficient in practice
- refresh operations reveal missing freshness controls
- post-launch learning reveals that support-strength scoring was too crude

Version updates should record what governance weakness prompted the revision.

---

## 30. Final Standard Statement

The Market-Intelligence Operating System is only as strong as its evidence discipline.

The Evidence Ledger exists to ensure that the system’s outputs are not merely persuasive artifacts, but governed judgments with traceable foundations, visible uncertainty, and refreshable integrity.

Without T11, the operating system can still produce intelligence.
With T11, it can produce intelligence that survives scrutiny.
