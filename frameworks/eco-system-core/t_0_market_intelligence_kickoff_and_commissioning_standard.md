## 1. Purpose

The **Market-Intelligence Kickoff & Commissioning Standard** is the formal intake and initiation layer for the Market-Intelligence Operating System.

Its function is to ensure that every ecosystem study begins with sufficient structure, scope definition, decision context, boundary discipline, and evidence expectations before the analytical workflow starts.

This document exists because the operating system can only produce strong conclusions if the input problem is framed correctly. Without a formal kickoff layer, ecosystem studies risk starting with:

- vague ecosystem boundaries
- unclear decision objectives
- missing time horizons
- hidden assumptions
- mismatched evidence expectations
- premature product thinking
- inconsistent output quality across studies

T0 exists to prevent those failures by standardizing how the system is commissioned.

---

## 2. Strategic Role in the Operating System

The Market-Intelligence Operating System already contains:

- the foundational ecosystem analysis doctrine
- the intelligence suite
- the strategy suite
- the T1–T19 standalone standards

What it still needs is a formal front door.

T0 serves ten strategic roles:

1. **Commissioning discipline** — ensures every analysis begins from a defined brief rather than a loose prompt.
2. **Boundary control** — defines what ecosystem is actually being studied and what is excluded.
3. **Decision alignment** — forces clarity on what decision the analysis is meant to support.
4. **Input sufficiency** — ensures the system has enough context to generate meaningful conclusions.
5. **Workflow routing** — determines which documents in the operating system are required versus conditional for a given study.
6. **Evidence readiness** — clarifies expected source classes, available evidence, and known information gaps before analysis begins.
7. **Anti-drift protection** — prevents premature collapse into product ideation, features, or tactical execution.
8. **Comparability support** — makes multiple ecosystem studies easier to compare later under T18.
9. **Agent readiness** — creates a standardized template that humans or agents can fill consistently.
10. **Quality gating** — establishes whether the study is ready to begin or whether more scoping is needed first.

T0 is therefore the formal commissioning layer for the full operating system.

## 3. What This Document Is

This document is:

- the official kickoff and intake standard for the operating system
- a commissioning brief template
- a readiness gate before Ecosystem Analysis begins
- a workflow-routing standard that determines which downstream documents should be produced
- the canonical starting point for each new ecosystem study

## 4. What This Document Is Not

The Kickoff & Commissioning Standard is **not**:

- the Ecosystem Analysis itself
- a product brief
- a feature ideation document
- a market-size memo
- a general brainstorm worksheet
- a substitute for T11 evidence governance or T12 uncertainty governance

It is the structured setup layer that enables all later documents to be produced with rigor.

---

## 5. Core Recommendation

Yes — the operating system should have a formal kickoff template standard.

Without T0, each ecosystem study must be improvised at the point of entry. That weakens consistency, increases analytical drift, and makes it harder to know whether conclusions are strong because the system worked or weak because the study began with poor framing.

With T0, every ecosystem analysis starts from:

- a defined ecosystem specification
- a clear decision question
- declared boundaries
- known evidence conditions
- a document-routing plan
- explicit readiness criteria

This should be treated as mandatory for all formal studies.

## 7. When to Use T0

T0 should be used whenever:

- a new ecosystem is being introduced into the system
- a new platform ecosystem, market ecosystem, or vertical is being studied
- a prior ecosystem study is being substantially reframed
- an agent or researcher needs a standardized intake packet before analysis begins
- multiple ecosystems will later be compared, prioritized, or routed into the same operating system

T0 should be completed before the Ecosystem Analysis is started.

## 8. Outputs of T0

A completed T0 should produce the following outputs:

- a clear ecosystem specification
- a clearly stated governing decision question
- a scope boundary definition
- a declared time horizon
- a known-input inventory
- a known-gap inventory
- a routing decision for which downstream documents are mandatory, recommended, or optional
- a readiness judgment on whether the study can begin immediately
- a canonical kickoff packet that can be handed to a researcher or agent

## 9. T0 Governing Questions

A good kickoff process should answer the following before analysis begins:

1. What ecosystem exactly are we studying?
2. What decision is this study meant to support?
3. What is inside scope and outside scope?
4. What geography, platform, segment, or category boundaries apply?
5. What time horizon matters?
6. What inputs do we already have?
7. What information is missing but important?
8. Which downstream documents are required for this case?
9. What would make the study insufficient before it even begins?
10. What must remain explicitly out of bounds to avoid premature ideation?

## 10. Required Structure of the Standard

Every T0 commissioning packet should contain the following sections.

### 10.1 Section 1 — Analysis Request Header
This section defines the basic request metadata.

Required fields:
- study title
- ecosystem name
- commissioning date
- requested by
- owner / analyst / agent
- status
- version

### 10.2 Section 2 — Governing Decision Question
This section states the exact decision the study is meant to support.

Examples:
- Is this ecosystem strategically worth entering?
- What is the best entry path into this ecosystem?
- Is this platform ecosystem viable for a small independent developer?
- Which of several ecosystems should receive focus first?

This must be specific. A vague request produces vague analysis.

### 10.3 Section 3 — Ecosystem Specification
This section defines what the ecosystem is.

Required fields:
- ecosystem type
- primary domain
- core market or platform name
- key participant groups
- primary interfaces or surfaces
- primary channels or marketplaces
- relevant business model context
- one-paragraph ecosystem description

This section should describe the ecosystem itself, not a proposed product idea.

### 10.4 Section 4 — Scope Boundaries
This section defines what is inside and outside the study.

Required fields:
- geographic scope
- customer / participant scope
- platform scope
- adjacent categories included
- adjacent categories excluded
- timeframe of interest
- explicit exclusions

This section is critical for preventing analysis sprawl.

### 10.5 Section 5 — Decision Horizon and Strategic Context
This section defines the temporal and strategic lens of the study.

Required fields:
- decision horizon
- urgency level
- why this study matters now
- strategic objective supported
- level of rigor required

The same ecosystem may be studied differently depending on whether the goal is quick-profit exploration, long-term strategic positioning, or platform portfolio comparison.

### 10.6 Section 6 — Known Inputs and Existing Evidence
This section records what is already known before research begins.

Required fields:
- documents already available
- prior ecosystem knowledge
- existing audits, notes, or prior analyses
- connected platform or marketplace knowledge
- known players or actors already identified
- known relevant policies or constraints

This prevents the system from ignoring available context.

### 10.7 Section 7 — Known Unknowns and Open Questions
This section identifies what is already suspected to be unclear.

Required fields:
- top open questions
- areas of likely ambiguity
- terms needing clarification
- likely evidence gaps
- risk areas that may require special attention

This section does not replace T12. It is pre-analysis uncertainty framing.

### 10.8 Section 8 — Workflow Routing Matrix
This section determines which downstream documents should be produced.

Required outputs:
- Ecosystem Analysis Standard: mandatory or not
- T1–T5: mandatory / recommended / optional
- T6–T10: mandatory / recommended / optional
- T11–T19: mandatory / recommended / optional

Routing should be justified briefly.

### 10.9 Section 9 — Readiness Check
This section determines whether the analysis can begin.

Possible statuses:
- ready to begin
- ready with cautions
- not ready — more scoping required
- blocked — critical information missing

### 10.10 Section 10 — Anti-Drift Constraints
This section explicitly states what the study must not do prematurely.

Examples:
- do not propose products yet
- do not recommend features yet
- do not collapse into pricing or go-to-market tactics yet
- do not assume the ecosystem is attractive before analysis proves it

This keeps the operating system disciplined.

## 11. Workflow Routing Logic

T0 should include a routing framework that determines which downstream documents are needed for the specific study.

### 11.1 Always mandatory
For formal ecosystem studies, the following should generally be mandatory:
- Ecosystem Analysis Standard
- T10 Entry Vector Analysis, if the study is meant to support actual strategic entry

### 11.2 Usually mandatory for serious studies
These are generally required unless the ecosystem is unusually simple:
- T1 Culture Map
- T2 Lever Architecture
- T3 Participant Decision Architecture
- T5 Competitive Topology
- T6 Trust Architecture
- T8 Risk & Fragility Map
- T9 Capital Flow Atlas

### 11.3 Conditional based on ecosystem shape
These may be mandatory depending on context:
- T4 Language & Semiotics Map when communication fit matters materially
- T7 Temporal Rhythm Map when timing or seasonality matters materially
- T17 Platform Access & Constraint Audit when platform dependence is material
- T18 Portfolio Scorecard when multiple ecosystems or pathways will be compared

### 11.4 Mandatory when conversion and governance begin
These become required once the study is moving beyond pure analysis:
- T11 Evidence Ledger
- T12 Assumption & Unknowns Register
- T13 Executive Decision Memo
- T14 Vector-to-Concept Conversion Brief
- T15 Validation & Experiment Design Standard
- T16 Refresh & Monitoring Protocol
- T19 Learning Capture Memo

## 12. Minimum Required Inputs Before Starting

A study should not begin formally unless T0 can answer, at minimum:

- what ecosystem is being analyzed
- what decision the study supports
- what is inside scope
- what is outside scope
- what time horizon matters
- what the strategic objective is
- what is already known
- what is obviously missing

If those cannot be answered, the study is not ready.

---

## 13. Readiness Criteria

A study is **ready to begin** only if all of the following are true:

1. The ecosystem is defined clearly enough to distinguish it from adjacent spaces.
2. The governing decision question is explicit.
3. Scope boundaries are set.
4. Time horizon is stated.
5. Strategic objective is stated.
6. Known inputs are logged.
7. Known unknowns are acknowledged.
8. Routing decisions for downstream documents are made.
9. Anti-drift constraints are defined.

If these conditions are not met, the study should not advance into the Ecosystem Analysis Standard.

---

## 14. Required Usage Rules

The following rules are mandatory.

### Rule 1 — No analysis without a governing decision question
The system should not begin formal work on a vague ecosystem prompt.

### Rule 2 — No product ideation in T0
T0 is for scoping the ecosystem study, not proposing products.

### Rule 3 — No scope ambiguity
Inside-scope and outside-scope boundaries must be stated explicitly.

### Rule 4 — No hidden assumptions about why the ecosystem matters
The strategic objective and urgency must be stated directly.

### Rule 5 — No skipping routing
T0 must determine which downstream documents are required.

### Rule 6 — No starting with zero evidence awareness
Known inputs and known gaps must be logged before beginning.

### Rule 7 — No false readiness
If key scoping elements are missing, T0 must say the study is not ready.

## 15. Recommended Operating Procedure

### Step 1 — Define the ecosystem
State exactly what market, platform, or ecosystem is being studied.

### Step 2 — Define the decision
State exactly what decision the analysis must support.

### Step 3 — Draw the boundaries
Clarify geography, participants, platforms, time horizon, and exclusions.

### Step 4 — Log what is already known
Capture existing documents, known actors, existing platform knowledge, and prior findings.

### Step 5 — Log what is not known
Identify the major open questions and likely gaps.

### Step 6 — Route the workflow
Decide which of the Ecosystem Analysis, T1–T10, and T11–T19 documents will be required.

### Step 7 — Run the readiness check
Determine whether the study can begin now or needs more scoping first.

### Step 8 — Launch the analysis
Only after T0 is complete should the operating system proceed into Ecosystem Analysis.

## 16. Quality Standard / Acceptance Criteria

The Kickoff & Commissioning Standard should be considered acceptable only if all of the following are true:

1. The ecosystem is clearly specified.
2. The governing decision question is explicit.
3. Scope boundaries are defined.
4. Strategic objective and time horizon are defined.
5. Known inputs are logged.
6. Known unknowns are logged.
7. Workflow routing is completed.
8. Readiness status is explicit.
9. Anti-drift constraints are explicit.
10. The packet is clear enough that a new analyst or agent could begin the study without improvising the foundation.

---

## 17. Failure Modes and Misuse Warnings

Common failure modes include:

### 17.1 Vague ecosystem naming
Labeling the ecosystem too broadly or too narrowly to support useful analysis.

### 17.2 Decision ambiguity
Starting a study without knowing what decision the study is meant to inform.

### 17.3 Product contamination
Allowing the kickoff template to smuggle in preferred product directions before analysis.

### 17.4 Missing routing discipline
Failing to determine which downstream documents are required for the case.

### 17.5 Scope sprawl
Not defining what is excluded, leading to endless adjacent-market drift.

### 17.6 False readiness
Starting formal analysis even though key context is missing.

### 17.7 Agent inconsistency
Using freeform prompts instead of a standard intake packet, causing uneven quality across ecosystem studies.

## 20. Relationship to Agentic Research Systems

T0 should be the standard packet used whenever an agent is instructed to begin a new ecosystem study.

This means:

- the fields should be explicit and machine-legible where possible
- the routing matrix should make downstream document selection deterministic
- readiness status should be explicit rather than implied
- anti-drift constraints should be visible so agents do not jump prematurely into ideation

An agent should not begin a formal ecosystem study without a completed or near-complete T0 packet.

---

## 21. Versioning Notes

The Kickoff & Commissioning Standard should evolve when any of the following occur:

- repeated studies fail because of poor scoping rather than poor analysis
- routing logic proves too vague or too rigid
- analysts or agents repeatedly begin work without sufficient inputs
- portfolio comparison reveals inconsistent study setup quality
- T19 learning shows that early framing errors repeatedly distort later conclusions

Version updates should record what commissioning weakness prompted the revision.

---

## 22. Final Standard Statement

A rigorous analysis system needs a rigorous beginning.

The Market-Intelligence Kickoff & Commissioning Standard exists to ensure that every ecosystem study begins with the necessary clarity, boundaries, evidence awareness, and workflow routing to support strong downstream conclusions.

Without T0, the operating system depends on improvisation at the point of entry.
With T0, it begins from discipline.
