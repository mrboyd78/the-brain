## 4. Identified Challenges, Proposed Solutions, and Future Research Avenues

### Challenge: The "Iframe CSP Jail" Execution Blockade
When an ISV attempts to build a massive React Activity (Embedded App SDK) to run inside the Discord Voice Channel UI, they frequently rely on dynamic external libraries, video streaming payloads, and complex 3rd-party authentication structures. Discord operates these Activities within a terrifyingly strict iframe Sandbox controlled by draconian Content Security Policy (CSP) mathematics. If the ISV React application attempts to execute `window.open()`, inject unvetted inline CSS, access the user's local microphone without routing through the SDK, or pull data from an unauthorized external API, the Discord Chromium environment will brutally and silently annihilate the script, leaving the 50-person user base staring at a blank, unbroken white rectangle.
*   **Proposed Resolution:** "Strict Proxy Compliance and Deep SDK Binding." An enterprise ISV must fundamentally sever their reliance on standard "Wild West" web development logic. The entire application architecture MUST be pre-compiled explicitly obeying Discord CSP parameters. More crucially, all external network requests to ISV infrastructure MUST be physically mapped and routed entirely through the official `discord-sdk.commands.fetch()` proxy pathways. Any attempt to construct a standard OAuth pipeline will trigger strict Cross-Origin mapping failures; authentication must be executed relying entirely on the native token exchange mechanics provided explicitly by the host execution wrapper. Conquering the internal architectural sandbox is the final barrier to achieving 100% platform domination.
